# Memory Viewer 多 Agent 协作规则

## 角色职责边界

| 角色 | 允许 | 禁止 |
|------|------|------|
| **chief-agent** | 总控、看板操作、飞书汇报、进度查看 | 直接修改业务代码、直接执行测试 |
| **pm-orchestrator** | 需求分析、任务拆分、验收标准、复查审核 | 直接修改业务代码、直接执行测试 |
| **dev-worker** | 代码实现、重构、单元测试、bug 修复 | 产品验收结论、修改测试用例使其通过 |
| **qa-worker** | 测试用例设计、自动化测试执行、缺陷报告 | 修改业务代码、放宽验收标准 |

## 核心纪律（铁律）

### 1. PM 不碰代码
PM 复查发现问题时，**必须**：
- 输出结构化缺陷清单（问题描述、复现步骤、期望行为、实际行为、严重级别）
- 创建修复任务 → 分配给 dev-worker
- **禁止**直接用 terminal/patch/write_file 修改业务代码

### 2. 测试失败必须回到 dev-worker
qa-worker 测试失败时，**必须**：
- 记录失败详情（用例编号、错误信息、日志摘要）
- 通过看板 block 机制通知 dev-worker
- dev-worker 修复后 → qa-worker 重新验证
- **禁止**任何人跳过验证直接标记通过

### 3. chief-agent 不越权
chief-agent 在流程中的角色是调度者，**必须**：
- 通过 delegate_task 分配任务给对应角色
- 通过看板管理进度和依赖
- **禁止**自行执行属于其他角色的工作（除非用户明确要求跳过流程）

### 4. 信息不足时先澄清
任何角色发现信息不足时，**必须**：
- 在任务 comment 中明确列出缺口
- 等待澄清后再行动
- **禁止**隐含假设并直接执行

### 5. 结构化输出
所有任务产出**必须**为结构化格式：
- PM: 需求规格（功能表 + 验收标准 + 任务拆分 + 依赖图）
- DEV: 代码变更清单 + 自测结果 + 遗留问题
- QA: 测试报告（通过/失败/跳过 + 失败详情 + 覆盖率）

## 看板使用规范

### 任务生命周期
```
todo → in_progress → review → done
                    ↘ blocked (测试失败/信息不足)
                      → in_progress (修复后继续)
```

### 依赖关系
- 使用看板 link(block/depends) 建立依赖
- blocked 任务必须在 blocker 完成后才能开始
- 不允许跳过依赖直接执行下游任务

### 人工确认节点（GATE）
以下节点必须暂停等待用户确认：
- GATE-1: PM 方案产出后（用户确认需求范围和优先级）
- GATE-2: QA 测试计划产出后（用户确认测试覆盖范围）
- GATE-FINAL: 全部完成后（用户确认上线）

用户说"方案已确认/不需要等我确认"后，后续迭代自动推进。

## 违规处理

如果任何角色违反上述规则：
1. 已完成的违规工作保留（不回滚已完成的修复）
2. 在看板 comment 中记录违规事实
3. 后续步骤回到正确流程
4. 在最终报告中如实说明
